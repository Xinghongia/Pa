## 1. TCP/IP协议栈 - 数据的传输链路

### 1.1 四层模型

```
┌─────────────────────────────────────────────────────────────┐
│                     应用层 (HTTP/FTP/SMTP)                  │
│                    应用数据                                  │
├─────────────────────────────────────────────────────────────┤
│                     传输层 (TCP/UDP)                        │
│                      端口号                                 │
├─────────────────────────────────────────────────────────────┤
│                     网络层 (IP)                              │
│                      IP地址                                 │
├─────────────────────────────────────────────────────────────┤
│                   数据链路层 (以太网)                        │
│                    MAC地址                                  │
└─────────────────────────────────────────────────────────────┘

类比快递：
├── 应用层 = 寄件人/收件人信息（你寄什么、收什么）
├── 传输层 = 快递公司（顺丰/中通，决定用什么服务）
├── 网络层 = 物流网络（经过哪些中转站）
└── 数据链路层 = 最后一公里（快递员派送）
```

### 1.2 TCP三次握手 - 建立可靠连接

**为什么需要握手？**

```
发送请求到服务器
     ↓
服务器收到了吗？
     ↓
需要确认！
```

**三次握手过程**：

```
客户端                                    服务器
   │                                        │
   │ ──────── SYN=1, seq=x ──────────────→ │  第一次：客户端说"我要连接"
   │                                        │
   │ ←──── SYN=1, ACK=1, seq=y, ack=x+1 ── │  第二次：服务器说"好，我收
   │                                        │       到，你也准备好"
   │                                        │
   │ ──────── ACK=1, seq=x+1, ack=y+1 ───→ │  第三次：客户端说"确认，收
   │                                        │       到你的确认"
   │                                        │
   │           【TCP连接建立完成】             │
   │                                        │
```

**每个SYN的作用**：

| 次数 | 发送方 | 作用 |
|------|--------|------|
| 第1次 | 客户端 | 请求建立连接，说"我的序号是x" |
| 第2次 | 服务器 | 同意连接，说"我的序号是y，收到x" |
| 第3次 | 客户端 | 确认，说"收到y，开始传数据" |

**为什么是3次？不是2次或4次？**

```
❌ 2次不够：
客户端：我发请求了（服务器收到了吗？）
服务器：收到了（客户端收到了吗？）
→ 客户端没确认服务器收到了，客户端不知道服务器能不能收

✅ 3次刚刚好：
客户端：我发请求了
服务器：收到了，我也能收
客户端：确认
→ 双方都确认对方能收也能发

❌ 4次浪费：3次已经够了，4次多此一举
```

**爬虫中的应用**：

```
爬虫请求 https://example.com
     ↓
TCP三次握手（建立连接）
     ↓
HTTP请求（真正开始传数据）
     ↓
TCP四次挥手（断开连接）

【连接池的作用】
每次请求都要握手？ → 慢！
连接池复用连接？   → 快！（省去握手时间）
```

---

### 1.3 TCP四次挥手 - 断开连接

**为什么不能像握手一样三次？**

```
握手时双方都是主动：
  客户端主动 → 服务器被动

挥手时可能双方都有数据在传：
  客户端说"我发完了" → 服务器可能还在发数据
  所以不能一起关，要各自确认
```

**四次挥手过程**：

```
客户端                                    服务器
   │                                        │
   │ ──────── FIN=1, seq=u ──────────────→ │  第一次：客户端说"我发完了"
   │                                        │
   │ ←─────── ACK=1, ack=u+1 ───────────── │  第二次：服务器说"收到，等
   │                                        │       我发完"
   │                                        │
   │         【客户端停止发送，等待】          │
   │                                        │
   │ ←─────── FIN=1, seq=w ─────────────── │  第三次：服务器说"我也发完
   │                                        │       了"
   │                                        │
   │ ──────── ACK=1, ack=w+1 ─────────────→ │  第四次：客户端说"确认断开"
   │                                        │
   │           【等待2MSL后关闭】             │
   │                                        │
```

**为什么要等2MSL？**

```
MSL = Maximum Segment Lifetime（最大报文生存时间）

等待2MSL的原因：
1. 确保服务器收到最后的ACK
2. 让所有报文在网络中消失，防止影响新连接
```

**爬虫中的应用**：

```
❌ 不复用连接：
请求1：握手→传数据→挥手
请求2：握手→传数据→挥手
请求3：握手→传数据→挥手
→ 每次都要握手+挥手，很慢！

✅ 连接池复用：
请求1：握手→传数据（保持连接）
请求2：            传数据（复用）
请求3：            传数据（复用）
请求完成后：       挥手
→ 省去多次握手+挥手
```

---

### 1.4 UDP vs TCP

| 对比 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接（握手） | 无连接（直接发） |
| 可靠性 | 可靠（丢包重传） | 不可靠（丢就丢了） |
| 速度 | 较慢 | 快 |
| 用途 | HTTP/HTTPS/FTP | DNS/视频/游戏 |
| 数据顺序 | 保证顺序 | 不保证顺序 |

**爬虫中的应用**：

```
HTTP请求 → 必须是TCP（可靠）
DNS查询  → 通常用UDP（快）
WebSocket → TCP（可靠长连接）
```

```
┌─────────────────────────────────────────────────────────────┐
│                       客户端（浏览器/爬虫）                  │
└─────────────────────────────────────────────────────────────┘
                              │
                              │ 1. 建立TCP连接
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      TCP三次握手                             │
└─────────────────────────────────────────────────────────────┘
                              │
                              │ 2. 发送HTTP请求
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  请求结构：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 请求行：GET /path HTTP/1.1                           │   │
│  │ 请求头：Host: example.com                            │   │
│  │        User-Agent: Mozilla/5.0                       │   │
│  │        Accept: text/html                             │   │
│  │        ...                                           │   │
│  │ 请求体：（POST请求时才有）                            │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
                              │
                              │ 3. 服务器处理
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                     服务器（处理请求）                       │
└─────────────────────────────────────────────────────────────┘
                              │
                              │ 4. 发送HTTP响应
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  响应结构：                                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 状态行：HTTP/1.1 200 OK                              │   │
│  │ 响应头：Content-Type: text/html                      │   │
│  │         Content-Length: 1234                         │   │
│  │         Set-Cookie: session=abc123                   │   │
│  │         ...                                         │   │
│  │ 响应体：<html>...</html>                             │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
                              │
                              │ 5. 关闭TCP连接（四次挥手）
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                       完成一次请求                          │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 HTTP请求结构详解

#### 请求行

```
GET /search?q=python HTTP/1.1
↑    ↑                  ↑
方法  路径              协议版本
```

**常见HTTP方法**：

| 方法 | 作用 | 示例 |
|------|------|------|
| GET | 获取资源 | 访问页面、API查询 |
| POST | 提交数据 | 登录、提交表单 |
| PUT | 更新资源 | 更新数据 |
| DELETE | 删除资源 | 删除数据 |
| HEAD | 只获取头部 | 检查资源是否存在 |
| OPTIONS | 查询支持的方法 | CORS预检 |

#### 请求头（Headers）

```
Host: www.example.com          # 目标主机（必填）
User-Agent: Mozilla/5.0...     # 浏览器标识
Accept: text/html,application/json  # 能接受的内容类型
Accept-Language: zh-CN,zh      # 能接受的语言
Accept-Encoding: gzip, deflate  # 能接受的压缩方式
Content-Type: application/json  # 请求体的格式
Content-Length: 234            # 请求体长度
Cookie: session=abc123         # Cookie认证
Referer: https://google.com    # 从哪来的（防盗链）
Origin: https://google.com     # 请求来源（CORS）
Authorization: Bearer xxx       # 认证信息
```

#### 请求体（Body）

```
GET请求：通常没有请求体（参数在URL中）

POST请求：
Content-Type: application/json
请求体：{"username": "test", "password": "123456"}

POST表单：
Content-Type: application/x-www-form-urlencoded
请求体：username=test&password=123456
```

### 2.3 HTTP响应结构详解

#### 状态行

```
HTTP/1.1 200 OK
↑        ↑    ↑
协议版本 状态码 状态文本
```

**状态码分类**：

| 范围 | 类型 | 说明 |
|------|------|------|
| 1xx | 信息 | 正在处理 |
| 2xx | 成功 | 请求成功 |
| 3xx | 重定向 | 需要进一步操作 |
| 4xx | 客户端错误 | 请求有问题 |
| 5xx | 服务器错误 | 服务器有问题 |

**常见状态码**：

| 状态码 | 含义 | 爬虫中的意义 |
|--------|------|-------------|
| 200 | OK | 成功获取数据 |
| 301 | 永久重定向 | URL变了，用新URL |
| 302 | 临时重定向 | 临时跳转 |
| 304 | 未修改 | 用缓存 |
| 400 | Bad Request | 请求格式错误 |
| 401 | Unauthorized | 需要认证 |
| 403 | Forbidden | 被禁止访问 |
| 404 | Not Found | 资源不存在 |
| 429 | Too Many Requests | 请求太频繁被限流 |
| 500 | Internal Server Error | 服务器出错 |
| 502 | Bad Gateway | 网关错误 |
| 503 | Service Unavailable | 服务不可用 |

#### 响应头

```
Content-Type: text/html; charset=utf-8  # 内容类型
Content-Length: 1234                     # 内容长度
Content-Encoding: gzip                  # 压缩方式
Set-Cookie: session=abc123              # 设置Cookie
Cache-Control: max-age=3600             # 缓存控制
Expires: Thu, 01 Jan 2025 00:00:00 GMT # 过期时间
Last-Modified: Wed, 01 Jan 2025 00:00:00 GMT  # 最后修改时间
ETag: "abc123"                          # 资源标识
Server: nginx/1.18.0                    # 服务器类型
X-Request-ID: uuid                     # 请求追踪ID
```

#### 响应体

```
HTML页面、JSON数据、图片、视频等
```

## 3. HTTPS加密原理

### 3.1 HTTP vs HTTPS

```
HTTP：明文传输，谁都能看到
HTTPS：密文传输，只有客户端和服务器能解密
```

**为什么需要HTTPS？**

```
❌ HTTP传输（明文）：
你 ←─────── 数据（谁都能偷看）───────→ 服务器
    黑客可以截获：账号、密码、Cookie等

✅ HTTPS传输（密文）：
你 ←────【加密数据】────→ 服务器
    黑客截获的是乱码，解不开
```

---

### 3.2 TLS/SSL加密原理

**核心思想：混合加密**

```
阶段1：非对称加密（慢，但安全）
  客户端获取服务器的公钥
  用公钥加密"对话密钥"发给服务器

阶段2：对称加密（快）
  双方用"对话密钥"加密后续通信
```

**TLS握手过程**（类比银行开户）：

```
客户端                                    服务器
   │                                        │
   │ 1. Client Hello                       │
   │    （支持哪些加密算法？）               │
   │ ─────────────────────────────────────→ │
   │                                        │
   │ 2. Server Hello + Certificate         │
   │    （我选择这些算法，这是我的证书）     │
   │ ←───────────────────────────────────── │
   │                                        │
   │ 3. 验证证书                            │
   │    （证书是CA机构签发的吗？）           │
   │    （证书上的域名对吗？）               │
   │                                        │
   │ 4. Pre-Master Secret                   │
   │    （用公钥加密的对话密钥）             │
   │ ─────────────────────────────────────→ │
   │                                        │
   │ 5. 生成会话密钥                        │
   │    （双方用算法计算出相同的密钥）        │
   │                                        │
   │ 6. 切换到对称加密                       │
   │    （后续通信都用会话密钥加密）          │
   │                                        │
```

**证书验证流程**：

```
1. 证书包含：
   - 网站域名
   - 公钥
   - 颁发机构（CA）
   - 有效期
   - 数字签名

2. 验证步骤：
   a. 证书是否在有效期内？
   b. 域名是否匹配？
   c. 颁发CA是否可信？
   d. 数字签名是否正确？

3. 证书链：
   根证书 → 中间证书 → 服务器证书
```

---

### 3.3 对称加密 vs 非对称加密

| 类型 | 算法 | 特点 | 用途 |
|------|------|------|------|
| 对称加密 | AES/DES | 同一密钥，快 | 大量数据加密 |
| 非对称加密 | RSA/ECC | 公钥加密，私钥解密，慢 | 密钥交换 |

**HTTPS中的实际使用**：

```
TLS握手：
非对称加密（慢，但安全） → 交换会话密钥

后续通信：
对称加密（快） → 加密实际数据
```

## 4. Cookie与Session

### 4.1 Cookie的工作原理

**Cookie = 服务器给客户端发的"通行证"**

```
服务器响应：
Set-Cookie: session=abc123; Path=/; HttpOnly

后续请求：
Cookie: session=abc123
```

**Cookie的完整生命周期**：

```
1. 客户端发送登录请求
        ↓
2. 服务器验证账号密码，创建Session
        ↓
3. 服务器在响应头设置Cookie
   Set-Cookie: session=abc123; HttpOnly; Secure
        ↓
4. 浏览器保存Cookie
        ↓
5. 后续请求自动带上Cookie
   Cookie: session=abc123
        ↓
6. 服务器验证Cookie，找到对应Session
        ↓
7. 识别用户身份，返回个性化内容
```

**Cookie的属性**：

| 属性 | 作用 |
|------|------|
| Name | Cookie名称 |
| Value | Cookie值 |
| Domain | 生效域名 |
| Path | 生效路径 |
| Expires/Max-Age | 过期时间 |
| HttpOnly | JS能否访问（防XSS） |
| Secure | 是否仅HTTPS传输 |
| SameSite | 跨站请求策略 |

---

### 4.2 Session的工作原理

**Session = 服务端存储的用户信息**

```
【服务端存储】
SessionID: abc123
{
  "user_id": 12345,
  "username": "test",
  "login_time": "2025-01-01 10:00:00",
  "expire_time": "2025-01-02 10:00:00"
}
```

**Session vs Cookie**：

| 对比 | Cookie | Session |
|------|--------|--------|
| 存储位置 | 客户端（浏览器） | 服务器端 |
| 安全性 | 较低（暴露在网络） | 较高（仅传输ID） |
| 存储量 | 小（<4KB） | 大（受服务器限制） |
| 性能 | 快（不占服务器资源） | 慢（占服务器内存） |

---

### 4.3 爬虫中的Cookie处理

**场景1：登录后爬取**

```python
# 方法1：手动管理Cookie
session = requests.Session()
session.post('/login', data={'username': 'test', 'password': '123'})
response = session.get('/protected')  # 自动带上Cookie
```

**场景2：处理Cookie过期**

```python
# 检测到登录失效，重新登录
if response.status_code == 401:
    session.post('/login', data={'username': 'test', 'password': '123'})
    response = session.get('/protected')
```

## 5. 认证机制

### 5.1 常见认证方式

#### Basic认证（基本认证）

```
请求头：
Authorization: Basic base64(username:password)

示例：
Authorization: Basic dGVzdDoxMjM0NTY=

解码后：test:123456
```

**特点**：简单但不安全，用户名密码每次请求都暴露

#### Bearer Token（令牌认证）

```
请求头：
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

用于OAuth2.0、API认证
```

#### JWT（JSON Web Token）

**JWT结构**：

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VyX2lkIjoxMjM0NSwidXNlcm5hbWUiOiJ0ZXN0IiwiZXhwIjoxNzA3MDAwMDAwfQ.abc123signature
↑                    ↑                                        ↑
Header           Payload                                Signature
```

**JWT三部分**：

| 部分 | 内容 | 说明 |
|------|------|------|
| Header | {"alg": "HS256", "typ": "JWT"} | 算法和类型 |
| Payload | {"user_id": 123, "exp": ...} | 用户数据 |
| Signature | HMAC签名 | 防篡改 |

**JWT验证流程**：

```
1. 用户登录，服务器返回JWT
        ↓
2. 客户端保存JWT
        ↓
3. 后续请求带JWT
   Authorization: Bearer xxx.yyy.zzz
        ↓
4. 服务器验证JWT签名和过期时间
        ↓
5. 从Payload中获取用户信息
```

## 6. 代理机制

### 6.1 代理的作用

```
【直连】
爬虫 → 服务器（暴露真实IP）

【使用代理】
爬虫 → 代理服务器 → 服务器（隐藏真实IP）
```

**代理在爬虫中的用途**：

1. **隐藏真实IP**：防止被封
2. **突破IP限制**：有些网站限制IP访问次数
3. **切换地区**：获取不同地区的内容
4. **负载均衡**：分散请求压力

---

### 6.2 HTTP代理

**工作原理**：

```
爬虫 → 代理服务器 → 目标服务器

请求格式（发给代理）：
GET http://example.com/page HTTP/1.1
Host: example.com
X-Forwarded-For: real_ip （可选，代理可能添加）

响应：
HTTP/1.1 200 OK
...
```

**代码示例**：

```python
proxies = {
    'http': 'http://代理IP:端口',
    'https': 'http://代理IP:端口'
}

response = requests.get(url, proxies=proxies)
```

---

### 6.3 HTTPS代理（CONNECT隧道）

**工作原理**：

```
1. 爬虫发送CONNECT请求
   CONNECT example.com:443 HTTP/1.1

2. 代理建立TCP连接到目标服务器

3. 代理返回
   HTTP/1.1 200 Connection Established

4. 之后所有数据都通过这个隧道传输

5. TLS握手在隧道内完成，代理看不到加密内容
```

---

### 6.4 代理类型对比

| 类型 | 协议 | 特点 |
|------|------|------|
| HTTP代理 | HTTP | 只支持HTTP协议 |
| HTTPS代理 | HTTP + CONNECT | 支持HTTPS |
| SOCKS代理 | SOCKS4/SOCKS5 | 支持任意协议，更快 |

## 7. WebSocket

### 7.1 WebSocket vs HTTP

| 对比 | HTTP | WebSocket |
|------|------|-----------|
| 连接方式 | 请求-响应 | 双向长连接 |
| 通信方式 | 客户端发请求，服务器才能响应 | 双方都可以主动发消息 |
| 实时性 | 低（轮询/长轮询） | 高（实时推送） |
| 资源消耗 | 每次请求都要建连 | 建连一次一直用 |
| 使用场景 | 网页浏览、API | 聊天、游戏、实时数据 |

---

### 7.2 WebSocket握手过程

```
1. 客户端发送HTTP请求（带升级头）
   GET /ws HTTP/1.1
   Host: example.com
   Upgrade: websocket
   Connection: Upgrade
   Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
   Sec-WebSocket-Version: 13

2. 服务器同意升级
   HTTP/1.1 101 Switching Protocols
   Upgrade: websocket
   Connection: Upgrade
   Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

3. 切换协议，之后就是WebSocket通信
   客户端 ←──────────────→ 服务器
```

**关键头部**：

| 头部 | 作用 |
|------|------|
| Upgrade | 升级协议 |
| Connection | 保持连接 |
| Sec-WebSocket-Key | 浏览器生成的随机key |
| Sec-WebSocket-Accept | 服务器用key计算出的验证 |

---

### 7.3 爬虫中的WebSocket

**场景**：有些网站用WebSocket传输数据（如实时行情、聊天消息）

**抓取方法**：

```python
# 使用websocket-client库
import websocket

def on_message(ws, message):
    print(f"收到消息: {message}")

def on_error(ws, error):
    print(f"错误: {error}")

ws = websocket.WebSocketApp(
    "wss://example.com/ws",
    on_message=on_message,
    on_error=on_error
)
ws.run_forever()
```

## 本章总结

### 核心概念速查

| 概念 | 一句话记住 |
|------|----------|
| TCP三次握手 | 确认双方都能收也能发 |
| TCP四次挥手 | 各自确认发完、收完再断开 |
| HTTP请求结构 | 请求行+请求头+请求体 |
| HTTP响应结构 | 状态行+响应头+响应体 |
| HTTPS | TLS加密的对称加密通信 |
| Cookie | 服务器发给客户端的"通行证" |
| Session | 服务端存储的用户信息 |
| JWT | 可以验证的令牌 |
| 代理 | 转发请求，隐藏真实IP |
| WebSocket | 双向长连接 |

### 爬虫中的应用场景

| 知识点 | 爬虫中的应用 |
|--------|-------------|
| TCP连接 | 连接池复用，省去握手时间 |
| HTTP请求 | 构造请求头、POST数据 |
| HTTPS | 处理证书验证、SSL问题 |
| Cookie | 登录态维持、绕过认证 |
| JWT | 解析token构造请求 |
| 代理 | 切换IP、防封禁 |
| WebSocket | 抓取实时数据 |
