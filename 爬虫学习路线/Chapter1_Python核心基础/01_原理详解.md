# 1.1 协程与异步IO（重点！）

## 1.1.1 从阻塞说起

**你之前写的爬虫是这样的：**

```python
import requests

urls = ["https://example.com/1", "https://example.com/2", "https://example.com/3"]

for url in urls:
    response = requests.get(url)  # 🚫 阻塞！必须等这个请求完成，才能发下一个
    print(response.text)
```

**问题在哪？**

```
时间轴：
0ms  ─────────────────────────────────────────────────────────────→ 3000ms

请求1:   [=====等待1000ms=====]
请求2:                                      [=====等待1000ms=====]
请求3:                                                        [=====等待1000ms=====]

总耗时：3000ms（串行执行）
```

每个请求要等1秒，3个请求就是3秒。如果你要爬10万个页面呢？

---

## 1.1.2 同步 vs 异步 对比

```
【同步模式 - 排队打电话】
你 → 打电话给A → 等A说完 → 挂断 → 打电话给B → 等B说完 → 挂断 → ...
      ↑ 在等待的过程中，你什么都做不了（阻塞）

【异步模式 - 委托秘书】
你 → 告诉秘书："帮我打这100个电话，谁先接就说一声"
      ↑ 你可以继续做其他事，秘书在后台处理所有电话

100个电话同时进行，谁先完成谁先处理
```

## 1.1.3 Python协程工作原理图解

```
┌─────────────────────────────────────────────────────────────────┐
│                        主线程（Main Thread）                      │
│                                                                  │
│  ┌──────────────┐                                               │
│  │  事件循环     │ ← 这是核心！它像 dispatcher（调度员）一样工作   │
│  │  Event Loop  │                                               │
│  └──────┬───────┘                                               │
│         │                                                        │
│  ┌──────┴───────┬───────────────┬───────────────┐              │
│  ↓              ↓               ↓               ↓              │
│ ┌────┐        ┌────┐         ┌────┐         ┌────┐             │
│ │协程1│        │协程2│         │协程3│         │协程4│  ...       │
│ │请求A│        │请求B│         │请求C│         │请求D│            │
│ └──┬─┘        └──┬─┘         └──┬─┘         └──┬─┘             │
│    ↓              ↓               ↓               ↓              │
└────┼──────────────┼───────────────┼───────────────┼──────────────┘
     ↓              ↓               ↓               ↓
     └──────────────┴───────────────┴───────────────┘
                              ↓
                    【网络/IO等待阶段】
                    （此时协程让出执行权）
                              ↓
     ┌──────────────┬───────────────┬───────────────┐
     ↓              ↓               ↓               ↓
  响应到达      响应到达         响应到达        响应到达
     ↓              ↓               ↓               ↓
  协程1唤醒      协程2唤醒       协程3唤醒       协程4唤醒
```

**关键概念**：
- **协程（Coroutine）**：你可以理解为"可以暂停和恢复的函数"
- **事件循环（Event Loop）**：整个系统的调度中心，决定哪个协程在什么时候执行
- **await**：等待一个IO操作时，自动让出CPU给其他协程

## 1.1.4 同步请求 vs 异步请求 时间对比

```
假设每个请求耗时：100ms
爬取100个页面：

【同步模式 - 串行执行】
请求1[100ms] → 请求2[100ms] → 请求3[100ms] → ... → 请求100[100ms]
|------------------------------------------------------------|
                        总耗时：100 × 100ms = 10000ms = 10秒
                        实际CPU等待时间：10秒（浪费）

【异步模式 - 并发执行】
所有请求同时发起（事件循环调度）
|-----------|
  100ms后，所有响应同时到达
                        总耗时：100ms ≈ 0.1秒
                        效率提升：100倍！
```

## 1.1.5 Python异步编程三剑客

```python
import asyncio        # 异步编程的核心库
import aiohttp        # 异步HTTP客户端（requests的异步版本）
import asyncio.run    # 运行异步程序
```

**最小的异步爬虫示例：**

```python
import asyncio
import aiohttp

async def fetch(session, url):
    """定义一个协程函数（async def）"""
    async with session.get(url) as response:  # await自动让出CPU
        return await response.text()

async def main():
    """主协程"""
    urls = [f"https://example.com/page{i}" for i in range(100)]

    async with aiohttp.ClientSession() as session:  # 创建会话
        tasks = [fetch(session, url) for url in urls]  # 创建100个任务
        responses = await asyncio.gather(*tasks)  # 并发执行所有任务
        print(f"✅ 抓取了 {len(responses)} 个页面")

# 运行异步程序
asyncio.run(main())
```

## 1.1.6 协程的"暂停-恢复"机制

这是理解异步最重要的地方：

```python
import asyncio

async def task1():
    print("A")           # 第1步：打印A
    await asyncio.sleep(0)  # 第2步：模拟IO等待，立即让出CPU
    print("C")           # 第4步：CPU回来后继续执行
    return "完成1"

async def task2():
    print("B")           # 第3步：CPU切换到task2，打印B
    await asyncio.sleep(0)
    print("D")           # 第5步：CPU切回task1后，再切到task2
    return "完成2"

async def main():
    results = await asyncio.gather(task1(), task2())
    #         ↑ 并发执行task1和task2

asyncio.run(main())
# 输出顺序：A B C D （任务1打印A后让出，任务2打印B，然后继续任务1）
```

**为什么要`await asyncio.sleep(0)`？**
- 模拟IO等待（网络请求、文件读写）
- `await` 时协程"卡住"，但CPU去执行其他协程
- IO完成后，事件循环通知协程"你醒了，可以继续了"

## 1.1.7 实战：协程 vs 同步 性能对比

```python
import time
import asyncio
import aiohttp
import requests

URLS = ["https://httpbin.org/delay/1" for _ in range(10)]  # 每个请求故意延迟1秒

# ====== 同步版本 ======
def sync_way():
    start = time.time()
    for url in URLS:
        requests.get(url)
    print(f"同步耗时: {time.time() - start:.2f}秒")

# ====== 异步版本 ======
async def async_way():
    start = time.time()
    async with aiohttp.ClientSession() as session:
        tasks = [session.get(url) for url in URLS]
        await asyncio.gather(*tasks)
    print(f"异步耗时: {time.time() - start:.2f}秒")

# 运行
sync_way()              # 同步：约10秒（串行）
asyncio.run(async_way()) # 异步：约1秒（并发）
```

**预期输出：**
```
同步耗时: 10.23秒
异步耗时: 1.05秒
```

# 1.2 连接池与HTTPAdapter（重点！）

## 1.2.1 为什么需要连接池？

**没有连接池的情况：**

```
每次请求都要：建立TCP连接 → 发送数据 → 接收数据 → 关闭连接
                ↑
            每次都要握手！耗时约50-100ms

10个请求 = 10次握手 = 500-1000ms浪费在握手上了！
```

**有连接池的情况：**

```
第一次请求：建立TCP连接 → 发送数据 → 接收数据 → 【保持连接不关闭】
第2-100次请求：复用已有连接 → 直接发送数据 → 接收数据 → 关闭连接
                      ↑
              复用！不需要每次都握手
```

---

## 1.2.2 requests库的连接池机制

```python
import requests

# requests默认就使用了连接池！
# Session对象维护了一个连接池（默认最大连接数：10）

session = requests.Session()

# 第一个请求：建立连接
session.get("https://example.com/page1")

# 第2-N个请求：复用上面的连接
session.get("https://example.com/page2")  # 复用
session.get("https://example.com/page3")  # 复用
session.get("https://example.com/page4")  # 复用
```

**原理图解：**

```
┌─────────────────────────────────────────────────────────┐
│                   Session 对象                           │
│                                                          │
│   ┌─────────────────────────────────────────────────┐    │
│   │              连接池 (Pool)                       │    │
│   │                                                 │    │
│   │   Connection 1: example.com:443 (已建立)        │    │
│   │   Connection 2: api.github.com:443 (已建立)    │    │
│   │   Connection 3: ...                             │    │
│   │                                                 │    │
│   │   最大连接数: 10 (可配置)                        │    │
│   │   最大保持连接: 100 (可配置)                     │    │
│   └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘

请求 example.com → 从池里找example.com的连接 → 找到就复用
请求 新域名.com   → 池里没有 → 建立新连接 → 加入池中
```

## 1.2.3 HTTPAdapter - 精细控制连接池

`HTTPAdapter` 是 `requests` 库的一个扩展机制，让你为不同的"协议+域名"配置不同的连接策略：

```python
import requests
from requests.adapters import HTTPAdapter

session = requests.Session()

# 默认适配器（全局）
session.mount("https://", HTTPAdapter(
    pool_connections=10,   # 维护多少个连接池（不同域名算不同池）
    pool_maxsize=20,       # 每个池最大连接数
    max_retries=3          # 最大重试次数
))

# 为特定域名定制（优先级更高）
session.mount("https://api.github.com", HTTPAdapter(
    pool_maxsize=50,       # github API需要更多连接
    max_retries=5
))

# 对于重要接口，使用更保守的重试策略
session.mount("https://payment.example.com", HTTPAdapter(
    pool_maxsize=5,
    max_retries=10,        # 支付接口要多重试几次
    pool_block=True        # 池满时阻塞，而不是创建新连接（更安全）
))
```

---

## 1.2.4 为什么要为不同域名配置不同策略？

```
┌────────────────────────────────────────────────────────────────┐
│                    不同网站的访问策略                            │
├──────────────────┬───────────────────────────────────────────────┤
│ 普通网站          │  pool_maxsize=10, max_retries=3             │
│                  │  通用策略，不需要特别对待                      │
├──────────────────┼───────────────────────────────────────────────┤
│ GitHub API        │  pool_maxsize=50, max_retries=5             │
│                  │  需要频繁调用，调高连接数                      │
├──────────────────┼───────────────────────────────────────────────┤
│ 支付/银行接口      │  pool_maxsize=5, max_retries=10, pool_block │
│                  │  资金安全，限制并发，重试多次                  │
├──────────────────┼───────────────────────────────────────────────┤
│ 爬虫高频请求       │  pool_maxsize=100, max_retries=3             │
│                  │  高并发场景，连接数拉满                        │
└──────────────────┴───────────────────────────────────────────────┘
```

## 1.2.5 实际应用：完整的爬虫连接池配置

```python
import requests
from requests.adapters import HTTPAdapter
from urllib3.util.retry import Retry

def create_session(max_retries=3, pool_maxsize=20):
    """
    创建一个配置好连接池的Session

    参数:
        max_retries: 最大重试次数（针对5xx错误和连接断开）
        pool_maxsize: 连接池最大连接数
    返回:
        Session对象
    """
    session = requests.Session()

    # 配置重试策略
    retry_strategy = Retry(
        total=max_retries,
        read=max_retries,
        connect=max_retries,
        backoff_factor=0.3,  # 重试间隔：0.3s, 0.6s, 1.2s...
        status_forcelist=[500, 502, 503, 504],  # 只有这些状态码才重试
        allowed_methods=["HEAD", "GET", "OPTIONS", "POST"]
    )

    # 挂载适配器
    adapter = HTTPAdapter(
        pool_connections=10,      # 最多缓存10个不同域名的连接池
        pool_maxsize=pool_maxsize, # 每个池最多20个连接
        max_retries=retry_strategy
    )

    session.mount("http://", adapter)
    session.mount("https://", adapter)

    # 设置默认请求头
    session.headers.update({
        "User-Agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36",
        "Accept": "text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8",
        "Accept-Language": "zh-CN,zh;q=0.9,en;q=0.8",
    })

    return session
```

# 1.3 上下文管理器

## 1.3.1 什么是上下文管理器？

**问题引入：**

```python
# 读取文件的错误写法
file = open("data.txt", "r", encoding="utf-8")
content = file.read()
file.close()  # 如果上面出错了，这行不会执行，文件没关闭！

# 正确写法（繁琐）
file = open("data.txt", "r", encoding="utf-8")
try:
    content = file.read()
finally:
    file.close()  # 无论如何都会执行
```

**Python的优雅解法：**

```python
# 使用 with 语句（上下文管理器）
with open("data.txt", "r", encoding="utf-8") as file:
    content = file.read()
# 自动调用 file.close()，无论是否出错
```

---

## 1.3.2 上下文管理器的原理

```python
# 等价于下面的写法

file = open("data.txt", "r", encoding="utf-8")
try:
    content = file.read()
finally:
    file.close()

# ======== 原理 ========

# with 语句执行的流程：
# 1. 调用 __enter__ 方法（获取资源）
# 2. 执行 with 块内的代码
# 3. 调用 __exit__ 方法（释放资源）
```

---

## 1.3.3 自定义上下文管理器

```python
class MyConnection:
    """模拟一个需要管理生命周期的连接"""

    def __enter__(self):
        print("🔌 建立连接...")
        self.conn = "连接对象"
        return self  # 返回的对象可以在 with 块内使用

    def __exit__(self, exc_type, exc_val, exc_tb):
        print("🔌 关闭连接...")
        self.conn = None
        # 如果返回 True，异常会被吞掉（不推荐）
        return False

# 使用
with MyConnection() as conn:
    print(f"使用连接: {conn.conn}")
print("连接已关闭")

# 输出：
# 🔌 建立连接...
# 使用连接: 连接对象
# 🔌 关闭连接...
```

**更简洁的写法（装饰器）：**

```python
from contextlib import contextmanager

@contextmanager
def my_connection():
    """用装饰器创建的上下文管理器"""
    print("🔌 建立连接...")
    conn = "连接对象"
    try:
        yield conn  # 相当于 __enter__ 返回值
    finally:
        print("🔌 关闭连接...")

with my_connection() as conn:
    print(f"使用连接: {conn}")

# 输出同上
```

# 1.4 生成器与迭代器

## 1.4.1 迭代器协议

**什么是迭代？**

```python
# 遍历列表（你熟悉的）
numbers = [1, 2, 3]
for num in numbers:
    print(num)

# 遍历字符串
for char in "abc":
    print(char)
```

**迭代器的定义：**

```python
# 一个对象如果实现了 __iter__ 和 __next__ 方法，就是迭代器
# __iter__ 返回迭代器本身
# __next__ 返回下一个元素，StopIteration 表示迭代结束

class MyIterator:
    def __init__(self, data):
        self.data = data
        self.index = 0

    def __iter__(self):
        return self

    def __next__(self):
        if self.index >= len(self.data):
            raise StopIteration
        value = self.data[self.index]
        self.index += 1
        return value

# 使用
it = MyIterator([10, 20, 30])
for x in it:
    print(x)  # 10, 20, 30
```

---

## 1.4.2 生成器（Generator）

**生成器是创建迭代器的简洁方式：**

```python
# 普通函数：返回一次就结束
def func():
    return [1, 2, 3]

# 生成器函数：可以暂停、恢复，逐步产出数据
def my_generator():
    yield 1      # 第一次调用，返回1并暂停
    yield 2      # 第二次调用，从暂停处继续，返回2并暂停
    yield 3      # 第三次调用，返回3，结束

# 使用
gen = my_generator()
print(next(gen))  # 1
print(next(gen))  # 2
print(next(gen))  # 3
print(next(gen))  # StopIteration 异常
```

---

## 1.4.3 生成器 vs 列表 - 内存对比

```python
# 列表：一次性把所有数据加载到内存
def get_numbers_list(n):
    return [i for i in range(n)]

# 生成器：按需生成，不占内存
def get_numbers_gen(n):
    for i in range(n):
        yield i

# 对比
import sys

list_data = list(range(1000000))
gen_data = get_numbers_gen(1000000)

print(sys.getsizeof(list_data))  # ~8MB（列表占用）
print(sys.getsizeof(gen_data))  # ~200 bytes（生成器只存状态）
```

**内存占用图解：**

```
【列表：100万个元素】
内存：████████████████████████████ 8MB
      [1][2][3][4][5]...[1000000]

【生成器：100万个元素】
内存：██ 200 bytes
      只存储：当前索引 = 500000
             函数内部变量 = {...}

当你 for x in gen 时：
第1次：生成1，内存占用很小
第2次：生成2，内存占用很小
... 每次只占一点点内存
```

## 1.4.4 爬虫中的生成器模式

**场景：爬取100万条数据**

```python
# ❌ 错误：把所有数据存列表，内存爆炸
def crawl_all_pages():
    results = []
    for url in all_urls:  # 100万个url
        data = fetch(url)
        results.append(data)  # 全部存内存！
    return results

# ✅ 正确：使用生成器，逐条处理
def crawl_pages():
    for url in all_urls:  # 100万个url
        data = fetch(url)
        yield data  # 每次只产出一条数据，不占内存

# 使用
for item in crawl_pages():  # 每次只处理一条
    process_item(item)      # 处理完就释放
    save_to_db(item)         # 边爬边存，不积压
```

**完整示例：**

```python
import requests

def generate_page_urls(base_url, total_pages):
    """生成器：按需生成URL，不一次性生成10万个URL字符串"""
    for page in range(1, total_pages + 1):
        yield f"{base_url}?page={page}"

def crawl_generator():
    """生成器爬虫：边爬边处理，不积压数据"""
    for url in generate_page_urls("https://example.com/api/items", 1000000):
        response = requests.get(url)
        if response.status_code == 200:
            items = response.json()
            for item in items:
                yield item  # 每爬到一条数据就yield出去

# 使用：逐条处理百万级数据，内存始终保持低位
count = 0
for item in crawl_generator():
    save_to_database(item)
    count += 1
    if count % 10000 == 0:
        print(f"已处理 {count} 条数据")
```

## 1.4.5 yield from - 生成器委托

```python
# 嵌套生成器
def inner():
    yield from [1, 2, 3]  # 委托给另一个生成器

def outer():
    yield from inner()    # 委托
    yield 4

for x in outer():
    print(x)  # 1, 2, 3, 4

# 等价于
def outer():
    for x in inner():
        yield x
    yield 4
```

import hmac

## 1.5.2 对称加密（AES）

```python
from Crypto.Cipher import AES
from Crypto.Util.Padding import pad, unpad
import base64

# AES加密（ECB模式，不推荐，但常见于老旧JS）
key = b"1234567890123456"  # 16字节密钥
data = "Hello World"

cipher = AES.new(key, AES.MODE_ECB)
padded_data = pad(data.encode(), AES.block_size)  # 填充到16字节倍数
encrypted = cipher.encrypt(padded_data)

print(f"AES加密: {base64.b64encode(encrypted).decode()}")

# 解密
decipher = AES.new(key, AES.MODE_ECB)
decrypted = unpad(decipher.decrypt(encrypted), AES.block_size)
print(f"AES解密: {decrypted.decode()}")
```

## 1.5.3 非对称加密（RSA）

```python
from Crypto.PublicKey import RSA
from Crypto.Cipher import PKCS1_v1_5
import base64

# 生成RSA密钥对
key = RSA.generate(2048)
private_key = key.export_key()
public_key = key.publickey().export_key()

# 加密（用公钥）
cipher = PKCS1_v1_5.new(RSA.import_key(public_key))
encrypted = cipher.encrypt(b"secret message")
print(f"RSA加密: {base64.b64encode(encrypted).decode()}")

# 解密（用私钥）
cipher = PKCS1_v1_5.new(RSA.import_key(private_key))
decrypted = cipher.decrypt(encrypted, sentinel=None)
print(f"RSA解密: {decrypted.decode()}")
```

## 1.5.6 爬虫中加密的实际应用

```python
# 场景：某网站登录需要发送加密密码
import hashlib
import hmac
import time

def generate_sign(params, secret_key):
    """
    模拟常见的sign生成算法
    很多网站的加密参数都是这样的套路
    """
    # 1. 排序参数
    sorted_params = sorted(params.items())
    # 2. 拼接字符串
    query_string = "&".join([f"{k}={v}" for k, v in sorted_params])
    # 3. 加盐签名
    sign = hmac.new(
        secret_key.encode(),
        query_string.encode(),
        hashlib.sha256
    ).hexdigest()
    return sign

params = {
    "username": "test",
    "password": "123456",
    "timestamp": str(int(time.time()))
}
sign = generate_sign(params, "your_secret_key")
print(f"生成的签名: {sign}")
```
